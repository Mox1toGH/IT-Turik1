# 📚 Інструкція для новичків: Як виконати Issue в проекті IT-Turik1

Ця інструкція призначена для учасників проекту з мінімальним досвідом розробки. Дотримуйтесь кроків по порядку.

---

## 1️⃣ Знайти Issues на GitHub

1. Перейти за посиланням: [https://github.com/Mox1toGH/IT-Turik1/issues](https://github.com/Mox1toGH/IT-Turik1/issues)
2. Подивитися список issues
3. **Фільтрувати по користувачеві**: Натиснути на фільтр → знайти issues присвоєні тобі (lilahhsa)
4. Прочитати 2-3 issues та вибрати те, яке здається простішим для виконання

> 💡 **Порада**: Обирай issues з назвами, що описують конкретну задачу (bug fix, додавання функції), а не питання.

---

## 2️⃣ Підготовка в ChatGPT: Розуміння Issue

1. Скопіювати текст обраного issue
2. Відкрити [ChatGPT](https://chat.openai.com)
3. Надіслати наступний промпт:

```
Будь ласка, розпиши мені детально:
1. У чому суть цього issue?
2. Що саме потрібно зробити?
3. Які файли можуть потребувати змін?
4. Які етапи розв'язання?

Ось текст issue:
[ВСТАВИТИ ТЕКСТ ISSUE СЮДИ]
```

4. Прочитати відповідь та записати собі основні моменти

---

## 3️⃣ Підготовка в ChatGPT: Промпт для Codex

1. На тій же розмові з ChatGPT надіслати нову частину:

```
Тепер напиши мені детальний промпт для GitHub Copilot (Codex), який допоможе мені виправити/реалізувати це issue.
Промпт повинен бути:
- Конкретним і детальним
- Містити опис проблеми
- Містити опис бажаного результату
- Давати приклади коду (якщо потрібно)

Як формат, напиши його у вигляді готового промпту, який я можу скопіювати і вставити в Codex.
```

2. Скопіювати отриманий промпт та збeрегти його для наступного кроку

---

## 4️⃣ Налаштування в VS Code: Вибір гілки Dev

1. Відкрити **VS Code**
2. В нижній частині вікна знайти назву поточної гілки (branch)
3. Натиснути на неї
4. У списку знайти **`dev`** та вибрати її
5. Почекати, поки гілка завантажиться

> ✅ **Перевірка**: Внизу зліва повинно написати `dev`

### Крок 4.1: Оновлення гілки Dev (Fetch)

1. Натиснути на іконку `⋯` (three dots) біля source-control -> changes
2. Вибрати **Fetch** 
3. Почекати, поки VS Code завантажить останні зміни з GitHub

> 💡 **Чому це важливо?** Fetch перевіряє, чи немає нових змін від інших учасників проекту. Це запобігає конфліктам під час роботи.

> ✅ **Перевірка**: Не повинно виникнути помилок, гілка повинна залишитися на `dev`

---

## 5️⃣ Створення нової гілки для роботи

1. В нижній частині VS Code натиснути на назву гілки (dev)
2. Натиснути на `+ Create new branch` (або `Create branch`)
3. Дати назву гілці за форматом: `fix/НАЗВА-ISSUE` або `feature/НАЗВА-ISSUE`
   
   **Приклади:**
   - `fix/errors-in-tests`
   - `feature/add-validation`
   - `fix/tournament-crashes`

4. Переконатися, що вона створена від гілки `dev`
5. Натиснути Enter

> ✅ **Перевірка**: Внизу зліва повинна бути видна ваша нова гілка

---

## 6️⃣ Робота з Codex в VS Code

1. Відкрити **Codex** в VS Code:

2. Створи новий чат

3. Вставити промпт, який ви підготували в ChatGPT

4. Дочекатися, поки Codex генеруватиме код

5. Читати пропозиції та застосовувати їх:
   - Якщо запропоноване розв'язання подобається — натисніть на нього
   - Codex допоможе вас направити через редагування файлів

> 💡 **Порада**: Якщо щось незрозуміле, запитайте більше деталей прямо в чаті Codex.

---

## 7️⃣ Git: Додавання змін та Комміт

### Крок 7.1: Додавання змін

1. Переходимо на вкладку **Source Control** 
2. Бачимо список змінених файлів
3. Натиснути `+` біля кожного файлу, щоб додати його в **Staging** (або натиснути `+` біля "Changes" щоб додати все)

> 📌 **Важливо**: Файли мають перейти з розділу `Changes` в розділ `Staged Changes`

### Крок 7.2: Створення Комміту

1. В полі введення напротивописом `Staged Changes` написати повідомлення комміту
   
   **Приклади:**
   - `Fix: исправить ошибку в тестах`
   - `Add: добавить валидацию для турнира`
   - `Fix: resolver issue with tournament registration`

   > 💡 **Або** натиснути іконку зірочок/блискавки коло поля введення — VS Code згенерує повідомлення автоматично за допомогою AI

2. Натиснути кнопку **Commit** (або `Ctrl+Enter`)

3. Якщо коммітів кілька, повторити для всіх змін

> ✅ **Перевірка**: У розділі `Staged Changes` не має залишитися файлів

---

## 8️⃣ Публікування гілки на GitHub

Наниснути кнопку **Publish Branch** яка зявиться на вкладці **Source Control** на місці кнопки **Commit**

> ✅ **Перевірка**: Гілка повинна з'явитися на GitHub в розділі Branches

---

## 9️⃣ Створення Pull Request на GitHub

1. Перейти на [https://github.com/Mox1toGH/IT-Turik1/pulls](https://github.com/Mox1toGH/IT-Turik1/pulls)

2. Натиснути **New pull request** (зелена кнопка)

3. **Налаштування PR:**
   - **Base**: вибрати `dev` (в який гілках вливати)
   - **Compare**: вибрати вашу гілку (та, з якої вливати)

4. Натиснути **Create pull request**

---

## 🔟 Назначення Reviewers та Assignees

### Вибір Code Reviewer:

1. На сторінці PR знайти розділ **Reviewers** (справа)
2. Натиснути на нього
3. Вибрати рецензентів:
   - **Mykhailo-Tr** (бажано) або інші учасники проекту
   - Можна вибрати кількох людей

### Назначення себе:

1. Знайти розділ **Assignees** (справа)
2. Натиснути на нього
3. Вибрати себе (lilahhsa) — **Assign yourself**

---

## 1️⃣1️⃣ Опис змін у PR Description

1. На сторінці PR знайти розділ **Description** (текстове поле внизу)

2. **Опція А**: Використати ChatGPT:
   - Скопіювати те, що вам дав Codex (або результат виконання)
   - Вставити в ChatGPT з проханням: "Напиши опис для Pull Request" з цих змін:
   ```
   [вставити опис змін]
   ```
   - Скопіювати отриманий опис та вставити в Description

3. **Опція Б**: Написати самостійно:
   - Описати що було зроблено
   - Які файли змінені
   - Як тестувати виправлення (якщо потрібно)


4. Натиснути **Update** або **Create**

---

## 1️⃣2️⃣ Повернення до гілки Dev (завершення)

1. Відкрити **VS Code**
2. В нижній частині натиснути на назву поточної гілки
3. Вибрати `dev` зі списку
4. Почекати завантаження

> ✅ **Готово!** Ви виконали всі кроки та створили Pull Request

---

## 🚀 Що далі?

1. **Чекайте рецензії**: Рецензент (Mykhailo-Tr) або інші переглянуть ваш PR
2. **Можливі коментарі**: Якщо буде що-небудь менеджменту, ви отримаєте коментарі в PR
3. **Правки**: Якщо потрібні виправлення, внесіть їх та зробіть новий комміт
4. **Merge**: Після схвалення рецензента PR буде влитий в `dev`

- [ ] Base: `dev`, Compare: ваша гілка
- [ ] Вибрані Reviewers
- [ ] Присвоєне Assignees (ви сами)
- [ ] Написаний опис змін
